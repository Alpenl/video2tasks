# 速度和质量主要由哪几个配置决定

这个项目配置很多，但真正决定“跑多久”和“切得好不好”的，主要是少数几个参数。

你可以用两个概念理解 Stage 1 的成本：

- 窗口数量：要发多少次请求。
- 单窗口成本：每次请求上传多少图、模型要推理多久。

Stage 1 总耗时可以粗略看成：

```text
总耗时 ~= 窗口数 * repeat 次数 * 单窗口平均耗时 / worker 并发数
```

其中“单窗口平均耗时”常常由模型端点排队/限流决定，所以会出现同配置不同视频差很多的情况。

## 主要影响“窗口数”的参数

### `windowing.window_sec` 和 `windowing.step_sec`

- `step_sec` 越小：窗口越密，窗口数越多，更慢；但边界更容易被“刚出现就捕捉到”。
- `step_sec` 越大：窗口越少，更快；但边界更可能延后，甚至被错过。

窗口重叠量是：`window_sec - step_sec`。

经验上，如果你觉得太慢：

- 优先考虑增大 `step_sec`（只要质量还能接受）。
- `window_sec` 不要过小，否则一个窗口里上下文不足，模型更容易乱切。

### `windowing.window_repeat_count`

这个参数会让同一批窗口重复跑多次再汇总。

- `1` 是常规。
- `2` 基本等价于 Stage 1 工作量翻倍。

除非你有明确证据证明 repeat 带来明显质量收益，否则不要开高。

## 主要影响“单窗口成本”的参数

### `windowing.frames_per_window`

它决定一个窗口里抽多少逻辑帧。

- 帧数多：时间覆盖更细，边界可能更准。
- 帧数多：联系图张数增加，或单张联系图更“拥挤”，上传更大，更慢。

### `windowing.contact_sheet_rows` / `windowing.contact_sheet_cols`

联系图每张能放 `rows * cols` 张帧。

- 网格越大：每张图塞的帧越多，单窗口需要的图片张数可能更少。
- 网格越大：每一格帧更小更糊，模型更难看清细节，可能影响质量。

建议做法：

- 如果模型经常漏掉很细的动作变化，考虑降低网格密度（更少 rows/cols）或提升分辨率。
- 如果瓶颈明显在上传/端点吞吐，才考虑提高网格密度来减少图片张数。

### `windowing.target_width` / `windowing.target_height`

分辨率越高：细节更清楚，但上传字节更多，模型推理也可能更慢。

在联系图模式下，这个参数影响很大，因为一张联系图包含很多帧。

### 图片编码（例如 `png_compression`、后端 JPEG 质量）

这组参数是在“本地 CPU”和“网络带宽/端点吞吐”之间做交换：

- 压缩强：文件更小，上传可能更快，但本地编码更慢。
- 压缩弱：本地更快，但上传更大，端点更慢。

如果你确认瓶颈在模型端点（最常见），减少上传字节往往更有效。

## 主要影响“并发”的参数

### `worker.count`

它决定一个样本内部并行处理窗口的数量。

增大 worker 的效果取决于端点是否限流：

- 端点承受得住：worker 越多越快。
- 端点承受不住：会出现 429/5xx，触发重试，反而更慢。

建议：

- 从一个中等值开始（例如 4 到 8），观察是否大量重试。
- 如果重试多，先降并发，再调其他参数。

## Stage 2（合并/字幕）为什么有时也会慢

Stage 2 通常不是瓶颈，但在以下情况下会变慢：

- 一次性把非常长的 segments 列表塞进一个请求。
- 让模型同时做太多事（合并 + 总结 + 翻译 + 校验），提示词很长。

一般的优化方向：

- 分块处理（先合并，再总结/翻译，或按段数分块）。
- 只让模型做文本任务，不要把 Stage 1 的帧布局信息带进 Stage 2。
- 让输出更“可校验”（严格 JSON、必须覆盖全部 seg_id、顺序不变），失败就 fallback。

## 常见“慢/不稳”的表现与原因

### 同样配置，有时 7 分钟，有时 17 分钟

这通常不是你本机慢，而是端点侧波动：

- 排队
- 限流
- 高峰期负载
- 少量窗口触发重试，拖出长尾

### 快结束时卡住

常见原因是“少数几个窗口变成拖尾”：

- 大部分窗口都完成了，但有几个一直在重试/超时。
- 体感上像是“最后一步特别慢”。

### 导出慢

多半是本地 ffmpeg 工作量：

- 段数太多（要切很多 clip）
- 输出分辨率太高
- 磁盘 IO 压力大

导出时间更接近随“段数”增长，而不是随“窗口数”增长。

## 一个安全的调参方式（避免乱试）

1. 先固定 Stage 2 和导出，只调 Stage 1。
2. 一次只改一个参数。
3. 每次都记录：窗口数、段数、总耗时、配置 diff。
4. 质量明显变差就立刻回退，不要叠加多个改动。
